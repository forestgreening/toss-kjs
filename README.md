# 경조사 마음장부 — 앱인토스 미니앱

> ✅ **앱인토스(Apps in Toss)에 출시 완료** — 스토어 노출명 '경조사 마음장부'
> React + Vite 기반 로컬 퍼스트 미니앱 · 도메인/저장소 테스트 102개 통과
> AI로 기획(멀티 에이전트 브레인스토밍·ralplan) → 개발 → 출시까지 완주한 프로젝트

---

## 0. 한 줄 정의

**"평생 경조사 주고받기 장부"** — 결혼·장례·돌·집들이·생일 등에서 내가 주고/받은 금액·선물을 빠르게 기록하고, 평생 품앗이(상호부조) 균형을 판단하게 돕는 **로컬 퍼스트** 앱인토스 미니앱.

청첩장 만들기·전파·송금 자동기록 기능은 **의도적으로 제거**했다(아래 4·5장 근거).

## 1. 배경 & 아이디어 진화 과정

처음 아이디어에서 시작해, 분석을 거치며 화려한 곁가지를 하나씩 걷어내고 단단한 코어만 남긴 과정.

| 단계 | 아이디어 | 분석 결과 |
|---|---|---|
| 출발 | 경조사 주고받은 금액·선물 기록 (연락처 기반) | 품앗이 문화에 맞는 검증된 수요 |
| +전파 | 내 경조사 발생 시 연락처로 전파(청첩장/부고) | 성장엔진처럼 보였으나… |
| +자동기록 | 토스페이 송금 시 양쪽 장부 자동기록 | …플랫폼/법에 막힘 (아래) |
| +수동 | 앱 안 쓰는 사람 위해 수동 기록 | **이게 사실상 코어**로 판명 |
| 청첩장 재검토 | "청첩장은 다른 데서 만든다" | 청첩장 빌더는 레드오션, 만들지 않기로 |
| 유통 재검토 | "만든 청첩장을 유통만" | 단톡방 일괄 공유라 하객 명단이 안 남음 → 유통도 약함 |
| **결론** | **곁가지 전부 제거, "주고받은 기록"이 본질** | 데이터는 "초대"가 아니라 **"주고/받는 순간"**에 생긴다 |

### 두 개의 핵심 페인
1. **(받는 쪽)** 결혼식/장례 직후 "누가 얼마 냈나" 정산 — 지금은 수기/엑셀로 고생.
2. **(주는 쪽)** 평생 장부 — 내가 누구에게 얼마 냈는지 기록 → 나중에 그 사람 경조사 때 적정 금액 판단.

## 2. 플랫폼 결정

> **앱인토스(Apps in Toss)로 싸게 검증 먼저 → 검증되면 독립 앱으로 확장.**

### 앱인토스 vs 직접 개발 비교

| 기준 | 앱인토스 | 직접 개발 |
|---|---|---|
| 연락처 접근 | △ (범위 불명, WebView 미지원 가능) | ✅ 전체 주소록 |
| 전파 자유 | 🔴 막힘 (동의한 내 유저만) | △ (공유시트 한정, 법 동일) |
| 받은돈 자동기록 | 🔴 불가 (송금 API/콜백 없음) | ⚠️ Android만, 정책 위험 |
| **유통/콜드스타트** | ✅ **토스 트래픽** | 🔴 **제로베이스** |
| 수익화 | △ (송금무료·IAP) | ✅ 자유(PG 필요) |
| 개발/운영 비용 | ✅ 낮음 | 🔴 높음 (iOS+Android+백엔드) |
| 1인 개발 적합 | ✅ | 🔴 |
| 법적 리스크 | 동일 | 동일 |

**판단**: 이 제품의 사망 1순위는 "저빈도 + 유통"인데 유통은 토스가 압도. 직접 개발의 기능 자유(전체 연락처·SMS 파싱)도 절반은 법/정책에 다시 막힘 → **검증은 토스에서 싸게, 확장은 독립으로.**

## 3. 멀티 에이전트 분석 요약 (4개 관점)

### ① 시장/요구사항
- 품앗이는 "갚아야 할 사회적 채무" → 기록 동기 강함. 한국 특화 수요.
- ⚠️ 핵심 미검증 가정 3개(플랫폼): 비동의 연락처 발송 가능?, 송금 콜백?, 로그인↔전화번호 매칭 합법성?

### ② 수익화 (난이도 3.5/5)
- 토스 P2P 송금 **무료** → 송금 수수료 모델 구조적 불가.
- **메인**: 맥락 커머스(답례품·화환·상품권) + 제휴 송객(상조/웨딩). 장부 데이터가 곧 답례 구매 파이프라인.
- **보조**: 프리미엄 구독(통계·백업·가족합산).
- ⛔ **부고엔 절대 과금 금지.** "트래픽 큰 순간(부고) ≠ 과금 순간(답례)" → 시간축 분리.

### ③ 기술 타당성 (난이도 4/5, 조건부 타당)
- 결제 = 디지털상품 **IAP뿐, P2P 송금 API 없음, 송금 콜백/웹훅 없음** → 자동기록 불가.
- 메시지 발송 = 푸시+인앱, **동의한 내 유저에게 세그먼트로만** → 연락처 전파 불가.
- `fetchContacts` 존재하나 **반환범위 불명 + WebView 미지원 가능성**(권한표상 RN 전용).
- 빌드 = 순수 Vite 아님. `granite.config.ts` + `@apps-in-toss/web-framework` → 번들 → **콘솔 업로드**.
- 로컬 저장 **영속 비보장**(앱삭제/기기변경 시 유실, 토스 FAQ 명시).
- `userKey`(로그인)·`getAnonymousKey`(기기변경에도 동일) → 백업 키로 사용 가능(서버 검증 전제).

### ④ 리스크/법무 (현재 설계 그대로면 REJECT → 피벗으로 해소)
- **C1**: 상대방 금액·연락처를 **본인 동의 없이 서버 저장** → 개인정보보호법 충돌. → **로컬 퍼스트로 해소.**
- **C2**: 연락처 대량 전파 → 정보통신망법 제50조. → **전파 제거 / 공유시트 위임으로 해소.**
- **부고 + 송금 UI** = 브랜드 자살. → **금액 UI 분리.**
- 자동연동 도달률 ≈ 0.25%(양쪽 다 유저여야) → "자동" 마케팅 금지.

## 4. 설계 원칙 (확정)

1. **로컬 퍼스트**: 제3자 금액·연락처는 기기 로컬(IndexedDB)에만. 서버엔 userKey/익명 카운터만. 제3자 평문 서버 전송 0건.
2. **전파/청첩장 만들기 없음** (레드오션 + 스팸 위법).
3. **송금 자동기록 없음** — 앱인토스에 송금 API/콜백 없음. 송금 딥링크(openURL)도 심사 금지. "자동" 표현 금지.
4. **수동 입력이 코어** — 3초 내 1건 입력 UX가 생명. 연락처는 편의 기능이며 **권한 없어도 100% 동작**.
5. **부고는 금액 UI 분리 + 절대 과금 금지.**
6. **심사 카피 가드**: "정산/송금" 금지 → "기록/장부/주고받은 내역". 카테고리는 금융 아닌 생활/유틸리티(가계부).
7. **백업 (현재: 수동 JSON / 설계: E2E 암호화 클라우드 — 결정 A)**: 로컬 영속 비보장 → **현재 MVP는 수동 JSON export/import**로 안전망 제공. 향후 단말에서 패스프레이즈로 암호화 후 서버엔 **암호문만** 저장하는 클라우드 백업 연동 예정(암호화 도메인 모듈은 구현·테스트 완료, 서버/UI 연동 미구현).

## 5. 데이터 모델 (요약)

```
Person { id, displayName, phoneE164?, phoneRaw?,
         status: CONTACT_PICKED|MANUAL|TOSS_LINKED,
         mergedFrom[], note?, createdAt, updatedAt }

Event  { id, type: WEDDING|FUNERAL|DOL|HOUSEWARMING|BIRTHDAY|OTHER,
         title, ownerSide: MINE|OTHERS, date, createdAt, updatedAt }

Record { id, eventId?, personId, direction: GIVEN|RECEIVED,
         amount?, giftName?, date, source: MANUAL, memo?,
         deletedAt?, createdAt, updatedAt }
```

- **병합**: `phoneE164` 정규화 키로 soft merge(hard delete 금지, mergedFrom 추적). 전화번호 없는 동명이인은 **자동병합 금지 + "같은 사람인가요?" 확인 제안**.
- **저장**: IndexedDB(Dexie). 사진 미저장(용량 리스크 회피).
- **삭제권**: 사용자 erasure(완전 삭제, export본 포함)와 내부 merge-tombstone(soft) 구분.

## 6. MVP 범위 (수용 기준 AC1~AC13, 전부 pass/fail)

| ID | 핵심 |
|---|---|
| AC1a/1b | 입력 탭 수 ≤4(결정적) / 실사용자 중앙값 ≤3초(Phase 3) |
| AC2 | 연락처 권한 거부 시 전 기능 100% 동작 |
| AC3/3b | 전화번호 soft merge 6케이스 / 동명이인 병합 제안 |
| AC4 | 이벤트 통계(합계·건수·평균·Top3) 즉시 재계산 (기준 기기) |
| AC5 | 사람 net + 적정 금액 힌트("가장 최근 RECEIVED 단일 금액") |
| AC6 | 로컬 복원 (PoC ② 결과에 조건부) |
| AC7 | 제3자 평문 서버 전송 0건 |
| AC8 | 부고 결제 진입점 0개 |
| AC9 | 게스트↔로그인 데이터 머지 |
| AC10 | 100건/50명 성능 (기준 기기) |
| AC11 | export→clear→import 머지그래프 100% 복원 (원본 id 보존, wipe-and-restore 전용) |
| AC14 | E2E 백업 봉투에 제3자 평문 0건 + 패스프레이즈 서버 미전송 *(암호화 도메인 모듈 구현·테스트 완료, 클라우드 업로드 UI 연동은 미구현)* |
| AC15 | 새 기기 로그인+패스프레이즈로 100% 복원, 틀린 패스프레이즈 실패 *(암복호 왕복 테스트 완료, 서버/UI 연동 미구현)* |
| AC12 | 카피 가드("정산/송금" 0건) + 송금 딥링크 0건 |
| AC13 | 사용자 삭제/전체 초기화 플로우 |

### 비범위 (Out of Scope)
청첩장/부고장 만들기·전파·공유링크, P2P 송금 자동기록, 송금 딥링크(openURL), 자체 SMS/알림톡 대량발송, 서버 제3자 **평문** 저장, 가족합산·구독결제, 실제 커머스 연동(placeholder만), 다국어. *(E2E 암호화 클라우드 백업은 결정 A로 In Scope로 이동)*

## 7. 로드맵 (Phase)

- **Phase 0 — 토스 샌드박스 실측 (코딩 전 차단형 게이트)**
  PoC 3항목 실측 + 결정표대로 go/no-go: ① fetchContacts WebView 지원/반환스키마, ② IndexedDB 용량·영속성(quota<5MB 또는 미생존 시 주저장소 탈락), ③ userKey/anonymousKey 안정성.
  ⛔ **글로벌 KILL/ABORT**: 영속 불가 + anonymousKey 미안정(+연락처 미지원·심사 제약) 확인 시 → 미니앱 중단, 독립앱 재검토.
- **Phase 1 — 도메인 코어 + 로컬 저장** (UI 없이 단위 테스트로 완성: 모델·정규화·merge·통계·hint·export/import)
- **Phase 2 — 3초 입력 + 8개 화면** (Quick Entry, 이벤트, 평생 장부, 사람 상세, 백업/복원, 설정/삭제, 수익화 placeholder)
- **Phase 3 — 가설 검증** (실사용자 5~10명 모의 정산 → 정산 완료율·WTP 측정)

## 8. 검증할 핵심 사업 가설

> "받는 쪽, **결혼식 직후 정산**이 실제로 돈 낼 만한 페인인가?"

- 측정: 정산 완료율 ≥70%(사용성), WTP ≥40%(방향 신호, 통계적 유효 아님), 입력 ≤3초/건, 힌트 유용 ≥60%.
- ⚠️ n=5~10은 통계적으로 무의미 → **방향 신호**로만. **S3.2 직후 파운더가 build/pivot/kill 결정.** '정산 완료율'은 사업성이 아니라 사용성을 검증함.
